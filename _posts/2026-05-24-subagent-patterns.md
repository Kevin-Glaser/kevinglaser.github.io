# Subagent 四种模式及架构演化路径

## 何时升级到多 Agent

**信号一：上下文管理挑战**

多个能力领域的专业知识无法舒适地塞进单一 prompt 时，需要策略性地分发上下文。当 Agent 的上下文窗口接近满载时，模型在任务完成上的表现会显著下降，进入 "dumb zone"（迟钝区）。

**信号二：分布式开发需求**

多个团队需要独立拥有和维护各自的 Agent 能力时，各团队可以独立迭代而不互相干扰。

***

## 单 Agent 的困境

一个典型的企业客服单 Agent 系统，System Prompt 可能堆积：

- 100行基础回复话术
- 500行产品手册
- 200行退货退款政策细则
- 150行物流查询接口文档
- 处理投诉升级、安抚客户情绪、生成工单...

Token 爆炸，模型注意力分散，每个领域的专业知识都在互相稀释。

***

## 四种核心设计模式

### 模式一：Sub-Agents（子代理委派）

**核心思想**：Supervisor Agent 充当协调者，将任务分解后委派给专门的 Sub-Agent。

**关键特征**：

- 上下文隔离能力强，每个 Sub-Agent 拥有独立上下文窗口
- Sub-Agent 通常无状态，专注于完成单次被委派任务
- 整体对话状态与流程控制由 Supervisor 统一维护
- 天然支持并行执行，多个 Sub-Agent 可同时展开工作
- 用户始终通过 Supervisor 间接沟通，不直接与 Sub-Agent 交互

**Claude Code 实现**：Explore Agent（只读搜索专家）、Plan Agent（架构设计专家）、General-purpose Agent（全能选手）。

**适用场景**：需要并行检索多个信息源、跨多个知识领域协同工作的研究系统。

**典型流程示例**：

```
1. LeadResearcher 分析查询、制定策略
2. 并行派出 3-5 个 SubAgent，各自己独立搜索
3. 每个 SubAgent 执行 3+ 个并行工具调用
4. CitationAgent 处理引用和来源归属
5. 结果汇聚回 LeadResearcher 综合输出
```

**成本**：约 15 倍 token 消耗换来 90% 研究时间缩短。

***

### 模式二：Skills（技能渐进式加载）

**核心思想**：通过 SKILL.md 文件实现能力的渐进式加载。Agent 一开始只知道技能名称和描述，判断需要时才加载完整指令。

**关键特征**：

- 单一 Agent 负责全部推理与执行
- 所有技能共享同一个上下文窗口
- 对话状态自然连续保留在同一个 Agent 内部
- 整体执行过程以顺序方式展开
- 用户始终与同一个 Agent 直接交互，交互路径最短

**Claude Code 实现**：

```
.claude/skills/
├── deploy/
│   └── SKILL.md
├── review-pr/
│   └── SKILL.md
└── database-migration/
    └── SKILL.md
```

**Sub-Agent 与 Skills 的本质区别**：

- Sub-Agent：独立上下文，适合大量信息过滤
- Skill：共享上下文，适合需要连贯对话的场景

***

### 模式三：Handoffs（交接式状态驱动切换）

**核心思想**：活跃的 Agent 根据对话状态动态切换，Agent A 完成阶段后通过 handoff() 将控制权传递给 Agent B。

**关键特征**：

- 上下文并非整体共享，可根据需要选择性传递
- 系统状态在 Agent 切换过程中被持续保存和传递
- 采用严格的顺序执行，不支持并行展开
- Agent 切换过程对用户透明

**Claude Code 实现（工程模式，非框架特性）**：依赖三大要素：

1. **明确的阶段状态（State）** — 需要显式定义流程阶段。
   - 例如：阶段一为信息收集（前台接待），阶段二为技术诊断，阶段三为执行与修复
2. **角色约束的 Agent 视角** — 每个阶段都有一个"角色约束的 Agent 视角"。
   - 每个阶段有明确的角色定义和可操作范围
3. **显式的阶段完成条件（Handoff Trigger）** — 这是 Handoffs 能稳定运行的核心。
   - 每个阶段必须有完成条件（Exit Criteria），否则就会"卡在阶段里出不来"

**示例**：

```
当前阶段：intake
规则：
- 只能提问
- 不要给解决方案
- 当信息完整时，明确声明：`进入 diagnosis 阶段`

Claude 输出：信息已收集完成，进入 diagnosis 阶段。
系统注入：当前阶段：diagnosis，你现在技术支持 Agent……
```

**适用场景**：具有明确阶段划分的流程型场景，如多阶段客服或工单系统。

***

### 模式四：Router（并行分发与合成）

**核心思想**：Router 对输入进行语义拆分与职责分流，并行分发给专业 Agent，最后统一合成响应。

**关键特征**：

- 极强的并行能力
- 清晰的职责边界
- 各处理分支彼此独立、上下文完全隔离
- 通常是无状态的

**Claude Code 实现（三种形态）**：

- 主 Agent 中的一段路由决策逻辑（最常见）
- 可调用的 Tool（Router-as-Tool）
- 轻量的 Sub-Agent（只负责分类，不负责执行）

**示例**：

```
用户提问：「我们的退货政策是什么？最近的销售数据如何？」

Router 分解：
├── 查询 1：退货政策 → 政策文档 Agent
├── 查询 2：销售数据 → 数据分析 Agent
└── 合成结果 → 统一回答
```

**适用场景**：跨多个知识域或数据源的查询，如企业知识库场景。

***

## 性能与成本的关键发现

- **95%** 的性能差异可归因于三个因素：Token 使用量（80%）、工具调用次数与模型选择（15%）、其余因素
- 选择更合适的模型带来的性能提升，往往超过单纯将 token 预算翻倍的效果
- 多 Agent 系统普遍存在约 **15 倍** 的 token 成本放大效应

**不适用多 Agent 的场景**：

- 需要所有 Agent 共享完整上下文的场景
- 强耦合且高度顺序化的工作流
- 难以并行拆解的编码任务
- 依赖 Agent 之间实时协调的系统

***

## 生产环境的关键挑战

1. **状态性复杂度** — Agent 在多轮对话中维持状态，微小失败会级联放大。应对策略：在每个 Agent 输出端设置检查点，验证输出质量再传递。
2. **非确定性调试** — Agent 的动态决策使得传统日志分析不够用。需要完整的生产链路追踪（Production Tracing）。
3. **部署复杂度** — Agent 可能正在执行中打断会导致不可预测行为。可考虑 Rainbow Deployment（渐进式部署）。
4. **同步瓶颈** — 当前大多数 SubAgent 是同步执行的。未来的方向是打通异步执行 + Agent 间消息通道。

***

## 架构演进路径

```
你的任务需要多 Agent 吗？
├─ 单一领域、工具 < 5 个、上下文 < 50K tokens
│  └─→ 不需要。单 Agent + 好的 prompt 即可
│
├─ 单一领域、但工具 > 10 个
│  └─→ Skills 模式
│
├─ 多领域、各领域需要独立上下文
│  └─→ Sub-Agents 模式
│
├─ 需要多步骤状态流转
│  └─→ Handoffs 模式
│
└─ 需要跨多个数据源并行查询
   └─→ Router 模式
```

**典型演进路径**：

1. **第一阶段**：单 Agent + Tools — 适合大多数初期场景
2. **第二阶段**：单 Agent + Skills — 工具数量增多、prompt 臃肿时
3. **第三阶段**：Supervisor + Sub-Agents — 不同领域需要独立上下文和专业知识时
4. **第四阶段**：混合架构 — Router 处理分类，Sub-Agent 处理并行研究，Handoff 处理顺序流程

***

## 黄金法则

1. **从单 Agent 开始** — 只在遇到明确瓶颈时才升级
2. **先加工具，再加 Agent** — Tools 是最小的扩展单位
3. **选对模型 > 堆更多 token** — 升级模型的效果超过翻倍预算
4. **上下文隔离是核心价值** — 多 Agent 的第一价值不是并行，是隔离
5. **Token 成本要求高价值任务** — 不是所有场景都值得多 Agent

***

## 总结

**最好的架构不是最复杂的架构，而是恰好满足需求的最简架构。**

Agent 的复杂度，不是靠"更聪明的模型"解决的，而是靠"更好的结构"消化的。架构选型永远是性能、成本和可控性的三角博弈。
